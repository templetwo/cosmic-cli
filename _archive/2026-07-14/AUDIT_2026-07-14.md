# Cosmic CLI — Project Audit

**Date:** 2026-07-14 · **Commit:** `578f9a6` (v0.5.0), origin in sync, tree clean
**Method:** 4 parallel review agents (security, correctness, packaging, tests) + local static analysis (ruff, bandit via uvx, pytest+coverage). Headline findings verified byte-exactly against source before publication.

---

## Verdict

The Grok path is real work. `StargazerAgent` + `tools.py` are genuinely well-hardened — path confinement that survives `..`/`~`/absolute escapes, automatic `.cosmicbak` backups before every mutation, syntax-check-and-auto-revert on Python edits, read-before-overwrite gating, thrash breakers. No `eval`/`exec`/`pickle`/`yaml.load` anywhere. No disabled TLS. This is the work of someone who thought about blast radius.

**The problem is everything standing next to it.** `OllamaStargazerAgent` is a drifted copy of that agent with the safety logic *not carried over* — it accepts `exec_mode="safe"` and then ignores it completely. `consciousness_assessment.py` ships a serialization bug that silently destroys its own data. The packaging leaks stale metadata and ships a module that crashes on import.

The single most important fact in this report: **`cosmic_cli/ollama_agent.py` will execute any shell command the local model emits, unconditionally, with no sandbox, no blocklist, and no working-directory confinement — including when the user explicitly asked for safe mode.**

---

## CRITICAL

### C1 — `exec_mode="safe"` is a no-op in the Ollama agent
**`cosmic_cli/ollama_agent.py:53`, `:240-241`** — *verified*

`exec_mode` is stored in `__init__` at line 53 and **never read again anywhere in the file** (`grep -n exec_mode` returns exactly two hits: the parameter and the assignment). `SHELL` actions run:

```python
result = subprocess.run(content, shell=True, capture_output=True, text=True, timeout=30)
```

No `DANGEROUS_SHELL` check. No confirmation. No `cwd=self.root` — it runs in whatever directory the process happens to be in, not `work_dir`.

**Failure scenario:** User runs the Ollama path in what they believe is safe mode. The local model emits `SHELL: rm -rf ~` — perhaps steered by content it read from an untrusted repo. It executes. The safe-mode flag the user passed did nothing.

Contrast `agents.py:858-863`, where the same action *is* gated (imperfectly — see H1) and *does* get `cwd=str(self.root)`. The guard exists. It just wasn't copied.

### C2 — Ollama `READ` opens any path on the filesystem
**`cosmic_cli/ollama_agent.py:252-254`** — *verified*

```python
elif action_type == 'READ':
    with open(content, 'r') as f:
        output = f.read()
```

Raw `open()` on the LLM-supplied string. No `normalize_user_path`, no root confinement — the guards that `tools.py` and `context.py` both implement correctly for the Grok path.

**Failure scenario:** Model emits `READ: ~/.ssh/id_rsa` or `READ: ../other-project/.env`. Contents are read into `context_memory` and re-sent to `self.ollama_url` on the next turn. Combine with M1 below (a project-local `.env` can silently override `OLLAMA_BASE_URL`) and that destination is attacker-controllable.

**Fix for C1+C2:** Either delete `ollama_agent.py`, or make it call the same `tools.py` primitives and the same `_run_shell` guard as `StargazerAgent`. Two independent implementations of the same safety-critical logic is the root cause here, not the individual bugs.

## HIGH

### H1 — Safe-mode shell blocklist is case-sensitive
**`cosmic_cli/agents.py:858-863`** — *verified*

```python
if any(d in cmd for d in DANGEROUS_SHELL):      # no .lower()
if re.search(r"\brm\b.*(-[a-zA-Z]*[rf]|--recursive)", cmd):   # no re.IGNORECASE
```

`RM -RF /` matches neither check. Neither does `rm -fR ~`. The regex only matches flag clusters ending in lowercase `r`/`f`.

**Fix:** `cmd_l = cmd.lower()` before the substring check; add `re.IGNORECASE` to the regex. One-line change, closes the trivial bypass.

### H2 — The blocklist only blocks destruction, not exfiltration
**`cosmic_cli/agents.py:50-65`** — *verified*

`DANGEROUS_SHELL` covers `rm -rf`, `sudo`, `mkfs`, `dd`, fork bombs, `git push --force`. It does not cover `curl … -d @~/.ssh/id_rsa`, `wget --post-file`, `find -delete`, `git clean -fdx`, `>` truncation of arbitrary files, or appends to `~/.ssh/authorized_keys`.

This matters more than a generic "blocklists are incomplete" complaint, because the agent's context includes **file contents it has READ** — so untrusted repo content can steer the next `SHELL:` action. That's a live indirect-prompt-injection → exfiltration path in the *default* mode.

**Fix:** A blocklist cannot close this. Move to an allowlist of command prefixes for safe mode, or require confirmation for any command containing a network verb. `principles.py` already states "BLAST RADIUS. Safe mode blocks destructive shell" as an invariant — the intent is right; the implementation doesn't reach it.

### H3 — Secrets flow to the cloud LLM and into permanent plaintext logs
**`context.py:84-108`, `agents.py:190-208`, `agents.py:162-173`**

`READ: .env` succeeds like any other file — there's no filename denylist (`tool_list` hides dotfiles from *listings* at `tools.py:71`, but that doesn't stop a direct READ). Contents are embedded in the next Grok prompt, sent to xAI, and if echoed into `FINISH`, written **unredacted** to `~/.cosmic_echo.jsonl` and the per-session `.jsonl` under `~/.cosmic-cli/sessions/`. Both persist indefinitely in cleartext.

**Fix:** Denylist `.env`, `*.pem`, `id_*`, `credentials*`, `.netrc` at the `read_file` layer. Add a redaction pass on anything matching a key-shaped pattern before it hits the echo/session logs.

### H4 — The consciousness log corrupts itself and silently discards all history
**`cosmic_cli/consciousness_assessment.py:485`** — *verified byte-exactly*

Source reads `f.write('\\n')` — that's an escaped backslash, so it writes the **two literal characters** `\` and `n`, not a newline.

**Failure scenario:** After the first append, `~/.cosmic_consciousness_log.jsonl` is one run-on line of concatenated JSON. `_load_history()` (lines 104-132) does `readlines()` + `json.loads(line)`, which raises `JSONDecodeError` on the blob — caught by a broad `except (IOError, json.JSONDecodeError)` wrapping the *entire* load. Result: every history load silently returns empty, forever. The feature has never worked and never complained.

**Fix:** `f.write('\n')`. Then narrow that except clause so the next corruption is loud instead of silent.

### H5 — No timeout on any xAI SDK call
**`agents.py:324-329`, `:347-352`, `review.py:178-181`**

`Client(...)` and `chat.create(...).sample()` have no timeout configured. The mission loop bounds step count and shell timeouts carefully — and then makes an unbounded blocking network call at its core. If xAI hangs, `cosmic-cli do` hangs forever with no output.

**Fix:** Pass an explicit timeout to the client constructor; wrap `.sample()` with a deadline.

### H6 — `main.py` has 0% test coverage
473 statements — the entire CLI surface, every command, all arg parsing — with **zero** tests. No test file imports `cosmic_cli.main` at all. Overall project coverage is **31%**.

## What's done well

Worth stating plainly, because the critical findings above are all *absences* rather than bad code:

- **`tools.py:16-44` `normalize_user_path`** resolves first, then checks `relative_to(root)` — correctly defeats `..`, `~`, and absolute escapes. There's even a comment recording the prior `lstrip('./')` footgun fixed in `5679634`. This is the right pattern.
- **Automatic backups before every mutation** (`.cosmicbak` via `shutil.copy2`), and `tool_edit` **re-compiles Python and auto-reverts on `SyntaxError`**. That's a self-healing guard most agentic CLIs don't have.
- **No `eval` / `exec` / `pickle` / `yaml.load` / `__import__`** anywhere in the shipped package.
- **No disabled TLS** anywhere; xAI goes through the official SDK.
- **`cosmic-cli run` requires explicit `click.confirm`** before executing a user-typed shell command.
- **`tools.py` (68%) and `review.py` (75%)** have real behavioral tests against `tmp_path` — the model the rest of the suite should follow.
- **The launcher chain is correct** — `./cosmic` → `cosmic_launcher.sh` → `cosmic_cli.main:cli` is consistent with the pyproject entry point.

## Recommended order of work

1. **`ollama_agent.py`** — delete it, or route it through `tools.py` + the shared `_run_shell` guard. Nothing else on this list is as sharp. *(C1, C2, M8)*
2. **Two one-line fixes:** lowercase the blocklist check + `re.IGNORECASE` (H1); `f.write('\n')` (H4).
3. **Timeouts on the xAI client** (H5) — a hung API currently hangs the whole CLI.
4. **Secret hygiene** (H3) — denylist `.env`/keys at the read layer, redact before the echo log.
5. **Packaging cleanup** — `rm setup.py COSMIC_SETUP.md requirements.txt test_api.py`, `git rm -r cosmic-cli/ --cached .coverage`, fix or delete `consciousness_demo.py`. *(M2, M3, M4, M11, M12)*
6. **Rethink safe mode as an allowlist** (H2) — the blocklist can't get there.
7. **Test `main.py`** (H6) — 473 statements at 0% is the largest single blind spot.

**Not urgent but worth deciding:** `consciousness_assessment.py` (~1000 lines) is unwired, has a data-destroying bug, and a broken integration point. Either wire it up and fix it, or move it to `_archive/` with the rest. Leaving broken code in the shipped package is how the Ollama drift happened in the first place.
